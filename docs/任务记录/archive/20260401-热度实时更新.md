## 老板原话

> 我们现在的这个热度更新啊，是定时更新。如果我们把它改造成根据点赞互动量，让帖子的热度变成实时更新，这个好弄吗？耗费资源的情况怎么样？我们能承受吗？
> 
> 好，那你按照我们开发 SOP 流程，把这个任务做一个改造。以后所有的帖子，只要出现热度变更了，让它就实时更新。

## 我的理解

当前热度更新是每天 4:30 AM 定时批量更新，老板要求改为实时更新（点赞/评论/分享时立即更新该帖子热度）。

## 具体需求

- [x] 点赞 API 增加实时热度计算
- [x] 评论 API 增加实时热度计算
- [x] 分享 API 增加实时热度计算
- [x] 创建工具函数 utils/calculate-heat.js（复用现有逻辑）
- [x] 定时任务改为兜底校准（防止数据不一致）

## 技术要求

**修改文件**：
- `src/routes/api.js`（点赞/评论/分享 API）
- `src/utils/calculate-heat.js`（新建工具函数）
- `scripts/update_heat_scores.js`（改为兜底校准）

**数据库变更**：无（posts 表已有 heat_score 字段）

## 关联影响

**直接影响**：
- 点赞/评论/分享 API 响应时间增加 <10ms
- 数据库写操作增加（每次互动多 1 次 UPDATE）

**间接影响**：
- 用户体验提升（热度实时可见）
- 定时任务负载降低（从全量更新改为校准）

## 预期效果

| 指标 | 改动前 | 改动后 |
|------|--------|--------|
| 热度更新延迟 | 24 小时 | <1 秒 |
| 单次互动耗时 | +0ms | +5-10ms |
| 日均 CPU 消耗 | 10 秒（一次性）| 1-2 秒（分散）|
| 用户体验 | 点赞后次日可见 | 点赞后立即可见 |
