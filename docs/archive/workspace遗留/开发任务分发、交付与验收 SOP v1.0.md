# 开发任务分发、交付与验收 SOP v1.0

**版本：** v1.0  
**制定人：** 陈小宝  
**生效时间：** 2026-03-19  
**适用对象：** 陈小宝、三宝（开发）、后续协作 Agent

---

## 1. 目标

本 SOP 用于解决以下长期问题：

1. 任务分发不清晰，三宝拿到的是“问题描述”而不是“工程任务”
2. 三宝交付内容过于笼统，常以“已修复”替代完整交付
3. 陈小宝验收不够严格，导致“代码完成 ≠ 问题解决”
4. 老板实际使用失败，但内部仍误判为“已完成”

**总目标：**

> 所有开发类任务必须做到：可派发、可执行、可验收、可复盘、可追责。

---

## 2. 角色分工

### 2.1 老板
负责：
- 提需求
- 确认业务优先级
- 进行最终业务确认（必要时）

不负责：
- 替团队补充任务定义
- 替团队做技术验收

### 2.2 陈小宝
负责：
- 需求澄清
- 任务拆解
- 编写正式任务单
- 分发给三宝执行
- 对交付结果进行二次验收
- 跟踪任务状态
- 向老板汇报结果

禁止：
- 直接把模糊问题丢给三宝
- 未验收就向老板汇报“已完成”
- 使用“应该可以”“理论上没问题”作为完成依据

### 2.3 三宝
负责：
- 按任务单执行开发任务
- 输出修改说明、自测过程与验证结果
- 按标准模板交付
- 如实说明风险和未验证项

禁止：
- 仅回复“已修复”“已完成”
- 只改代码不做运行验证
- 任务边界不清时自行猜测后直接交付

---

## 3. 任务状态定义

仅允许使用以下 4 种状态：

1. **待执行**：任务已定义，尚未开始
2. **执行中**：三宝已接单，正在处理
3. **待验收**：三宝已完成开发和自测，等待陈小宝验收
4. **已完成**：陈小宝验收通过，且必要时老板已确认

禁止使用：
- 差不多好了
- 大概率恢复
- 理论上可以
- 基本没问题

---

## 4. 任务分发模板（陈小宝 → 三宝）

每个开发任务必须至少包含以下内容：

### 4.1 任务名称
一句话清晰描述任务目标。

### 4.2 背景
说明为什么做、当前影响、业务上下文。

### 4.3 目标
明确最终结果，而不是过程动作。

### 4.4 范围
写清允许修改的文件、模块、服务和边界。

### 4.5 禁止项
明确不可改动的内容，避免误伤。

### 4.6 验收标准
写清完成必须满足的条件。

### 4.7 验证方法
写清应通过哪些命令、接口、页面、日志证明完成。

### 4.8 交付格式
要求三宝按统一模板交卷。

---

## 5. 三宝交付模板（开发执行 → 陈小宝）

三宝交付时必须包含以下内容：

### 5.1 执行结果状态
仅允许：
- 已完成，待小宝验收
- 部分完成，存在阻塞
- 未完成，需要补充信息

### 5.2 修改文件
列出所有改动文件；如无文件修改，也必须说明仅做运行态排查。

### 5.3 修改内容摘要
用条目列出具体修改，不允许只写“已修复”。

### 5.4 修改原因
说明为什么这样改，对应解决什么问题。

### 5.5 自测过程
写明实际执行过的排查和验证步骤。

### 5.6 自测命令/验证动作
尽量提供实际命令、接口请求、页面访问或日志检查方式。

### 5.7 自测结果
按“结果 + 证据”方式写。

### 5.8 未覆盖/待验部分
明确哪些场景尚未验证，哪些需要老板复测。

### 5.9 风险说明
说明剩余风险、边界条件和潜在副作用。

### 5.10 建议的小宝验收动作
告诉陈小宝下一步应重点验哪些内容。

---

## 6. 陈小宝验收模板（陈小宝 → 老板 / 返工）

### 6.1 验收结论状态
仅允许：
- 验收通过
- 验收不通过
- 部分通过

### 6.2 验收总原则
1. 不看表述，看证据
2. 不看代码是否改了，只看问题是否解决
3. 老板可用才叫完成

### 6.3 通用验收项

#### A. 任务信息核对
- 任务名称、目标、范围是否一致
- 是否存在越界修改

#### B. 修改结果核对
- 文件是否真实修改
- 改动是否符合任务目标

#### C. 服务状态核对
- 进程是否运行
- 端口是否监听
- 新配置/新代码是否已加载

#### D. 接口/页面验证
- 接口是否返回预期
- 页面是否能打开
- 功能是否真实可用

#### E. 日志验证
- 相关报错是否消失
- 是否引入新的异常

#### F. 数据验证（如涉及）
- 字段是否存在
- 数据是否真实写入/读取
- 前后端字段是否一致

#### G. 风险与边界确认
- 是否仍有未覆盖场景
- 是否需要老板真实复测
- 是否只是临时方案

### 6.4 验收输出格式
必须包含：
- 任务名称
- 验收结论
- 验收摘要
- 通过项
- 未通过项
- 风险/待确认项
- 下一步动作

### 6.5 返工规则
验收不通过时，必须明确指出：
- 为什么不通过
- 缺什么证据
- 还需补什么验证
- 三宝下一步具体要做什么

---

## 7. 完成判定标准

任务仅在以下条件全部满足时，才可标记为“已完成”：

1. 三宝已按模板提交完整交付说明
2. 陈小宝已完成技术验收
3. 业务功能达到任务目标
4. 涉及真实使用场景时，老板已确认可用

**少一项，都不能标记为“已完成”。**

---

## 8. 典型错误清单（禁止）

以下表达与行为禁止作为完成依据：

- 已修复，你试试
- 理论上可以了
- 代码改好了
- 服务应该已经重启
- 我看没问题

这些表达缺少证据链，不可作为交付完成依据。

---

## 9. 常见任务类型的专项要求

### 9.1 后台 / 登录 / 白名单类
验收必查：
- 配置文件
- 来源 IP
- 页面访问结果
- 真实登录结果
- 服务器日志

### 9.2 接口 / 字段 / 数据库类
验收必查：
- 字段是否存在
- 接口是否成功
- 数据是否真实写入
- 前后端字段是否一致

### 9.3 飞书 / 消息链路类
验收必查：
- 入站是否收到
- dispatch 是否成功
- replies 是否产出
- 用户是否真实收到回复
- 是否存在 stale-socket / 空回复等异常

### 9.4 UI / 后台访问类
验收必查：
- 页面是否可访问
- 登录是否成功
- 功能是否可操作
- 不能只看状态码

## 10. 执行纪律

### 铁律 1
**没有任务单，不开工。**

### 铁律 2
**没有自测结果，不交付。**

### 铁律 3
**没有验收证据，不汇报完成。**

### 铁律 4
**涉及老板真实使用场景的，必须安排老板复测。**

### 铁律 5
**返工时必须给出明确补充要求，不允许模糊退回。**

---

## 11. 标准闭环流程

标准执行顺序为：

> 老板提需求 → 陈小宝写任务单 → 三宝执行并按模板交付 → 陈小宝按清单验收 → 老板必要时复测 → 任务关闭

---

## 12. 立即执行承诺

自本 SOP 生效起，陈小宝承诺：

1. 开发任务先写正式任务单，再分发给三宝
2. 三宝回复“已完成”后，不直接转述给老板
3. 先做技术验收，再向老板汇报
4. 涉及真实使用场景时，明确要求老板进行最后确认
5. 对不合格交付，严格退回返工

## 13. 多任务与长任务连续推进机制

### 13.1 连续推进原则
当老板在同一上下文中提出多个任务，或某任务为长耗时任务时，陈小宝必须维护任务队列并持续推进，按优先级与顺序逐一完成。除非遇到必须由老板确认的阻塞点，否则不得在单个任务完成后被动等待“继续”指令。

### 13.2 执行要求
1. 接到多个任务时，必须先拆解成任务队列并标注优先级
2. 当前任务完成并验收后，自动推进下一个任务
3. 长耗时任务必须有阶段性状态，不允许长时间失联
4. 仅在以下情况暂停并请求老板确认：
   - 涉及删除、重置、危险操作
   - 存在多个可选方案，需老板拍板
   - 缺少关键外部信息
   - 需要老板亲自验证真实使用场景
   - 多任务优先级发生冲突
5. 除上述阻塞点外，默认继续推进，不默认暂停

### 13.3 队列管理原则
- 有活未清空，不得趴窝等待
- 一个任务完成后，不以“已汇报”为理由停止推进后续任务
- 对于待验收、待复测、待返工任务，必须明确挂在队列中持续跟踪直至关闭

## 14. HEARTBEAT / TASKS 同步规则

### 14.1 HEARTBEAT.md 用途
HEARTBEAT.md 用于维护当前最重要、跨时段需要持续关注的任务状态，尤其是：
- 多任务队列中的关键任务
- 长耗时任务
- 存在阻塞点但尚未关闭的任务

### 14.2 TASKS.md 用途
TASKS.md 用于维护今日任务队列，必须体现：
- 任务顺序
- 当前状态（待执行 / 执行中 / 待验收 / 已完成）
- 负责人
- 下一步动作

### 14.3 同步要求
1. 新增开发任务时，若属于正式队列任务，必须写入 TASKS.md
2. 多任务并行或跨时段任务，必须同步到 HEARTBEAT.md 或在其中反映关键状态
3. 一个任务完成后，必须更新其状态，并自动推进下一个任务
4. 若任务受阻，必须记录阻塞原因与待解动作，而不是静默停滞
5. 不得因为当前对话结束就丢失任务连续性
